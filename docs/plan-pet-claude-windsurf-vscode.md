# 企划案：将桌面小猪扩展至 Claude Code、Windsurf、VS Code 生态

> **优先级：低**  
> 本文档描述**方向与愿景**、**使用场景与必要性**，不包含技术落地方案与排期。在主线（Cursor Hooks + 本地宠物体验）稳定、文档与质量达标之前，本方向仅作储备讨论。

---

## 1. 背景与问题陈述

当前 Nixie 桌面小猪主要通过 **Cursor Hooks** 感知 Agent 活动，并将状态写入本地 `~/.nixie/state.json`，由 `nixie-pet` 统一呈现。许多用户的工作流并不只绑定在 Cursor 上：**终端里的 Claude Code**、**Windsurf 的 Cascade**、以及 **VS Code 及生态内的 AI 扩展**同样是日常「和 AI 一起写代码」的主战场。

若小猪长期仅与 Cursor 强绑定，会带来两点张力：

- **叙事上**：产品愿景是「陪伴你写代码、感知 AI 是否在干活」，而非「只做 Cursor 插件人格」。
- **体验上**：同一用户在多工具间切换时，小猪会「失明」或只能反映其中一端，削弱陪伴感与一致性。

本企划案讨论的是：**在保持本地优先与隐私边界的前提下**，是否值得把「感知面」逐步延伸到 Claude Code、Windsurf、以及 VS Code 相关场景，以及**为何值得做、为谁而做**——**不**在此定义具体实现与里程碑。

---

## 2. 愿景与方向

### 2.1 愿景（一句话）

**让同一只小猪，在用户选择的不同「AI 编程表面」上，都能以一致、可预期的方式反映「Agent 是否在忙、在做什么类型的事」**——在无法或不宜获取的细节上保持诚实降级，而不是假装有云端用量或全知视角。

### 2.2 方向原则（与现有产品一致）

- **本地优先**：默认不依赖公网拉配置、不把用户工作流细节上传到第三方；延续现有 `state.json` 式本地契约的思路。
- **与上游同视界**：只消费各产品**官方暴露**的钩子、事件或等价机制能覆盖的信息；不通过猎奇手段扩大数据面。
- **单线叙事**：展示层仍以**单一连贯状态**为主（与用户已讨论的多 Agent 策略一致）；多源并存时的合并与「结束提醒」可作为远期增强，而非本企划的前提。
- **渐进可选**：多平台支持应是**可选路径**（用户显式安装/配置对应适配），而非强制全员迁移。

---

## 3. 目标平台与角色（概念层）

| 方向 | 在愿景中的角色（概念） |
|------|------------------------|
| **Claude Code** | 终端优先、与 IDE 解耦的会话；官方提供 [Hooks](https://code.claude.com/docs/en/hooks) 等生命周期扩展点，与「stdin JSON、事件驱动」的模型和现有 `nixie-hook` 哲学接近，适合作为**非编辑器绑定**的第一类延伸场景。 |
| **Windsurf** | 独立 AI IDE，提供 [Cascade Hooks](https://docs.windsurf.com/windsurf/cascade/hooks) 等机制；适合服务**以 Windsurf 为主力**、同时希望小猪状态与 Cascade 同步的用户。 |
| **VS Code** | 生态最广，但 **AI 能力分散在多个扩展与模式**，未必存在与 Cursor 完全同构的单一 Hooks 规范；在愿景中更适合定位为**「按场景逐步覆盖」**（例如：若某条官方或社区路径能提供稳定事件，再讨论是否接入），而非「一个开关通吃全 VS Code」。 |

以上仅为**角色分工**，不代表承诺交付顺序或范围。

---

## 4. 使用场景与必要性

### 4.1 典型使用场景

1. **多工具并行**：用户在 Cursor 里改前端，在终端里用 Claude Code 跑批处理脚本或长任务；希望小猪至少能反映「**有一端** Agent 仍在工作」，而不是完全静默。
2. **团队/课程指定环境**：公司或教学环境统一使用 VS Code + 某 AI 扩展，或统一 Windsurf；小猪若仅支持 Cursor，则这部分用户**无法**获得同等陪伴体验。
3. **个人偏好迁移**：用户从 Cursor 迁到 Windsurf 或「终端 + Claude Code」为主；希望**同一套情绪与交互习惯**可以迁移，而不是换工具就换宠物逻辑。
4. **品牌与开源叙事**：「本地宠物 + 多源可选适配」有利于说明项目边界清晰、扩展是**插件式**的，而非与单一商业 IDE 深度耦合。

### 4.2 必要性（为何值得讨论，即便优先级低）

- **必要性（用户价值）**：降低「装了我们却只能在一个软件里看见反应」的挫败感，使愿景与真实工作流更一致。
- **必要性（产品一致性）**：若长期仅 Cursor，外部容易将 Nixie 误解为 Cursor 附属；多平台**意向**与**克制**（只走官方扩展点）有利于正确定位。
- **为何仍是低优先级**：每增加一类上游，都要面对**事件语义对齐、多写入者策略、测试与维护成本**；在 Cursor 路径的体验与文档尚未「足够好」之前，盲目铺开会稀释主线质量。

## 5. 明确不包含的内容（本企划范围外）

- 具体技术方案（二进制拆分、配置格式、合并算法、与 `pet.sock` 的交互细节等）。
- 排期、人力估算、版本号承诺。
- 对 VS Code 内部未公开 API 或不稳定行为的依赖。

---

## 6. 结论性表述（供路线图引用）

拓展小猪对 **Claude Code、Windsurf、VS Code** 的感知能力，在方向上与「本地、可选、与官方扩展点对齐」的愿景一致，能覆盖真实多工具工作流并强化产品叙事；但实现与维护成本显著，且依赖各平台公开能力的成熟度，故整体列为 **低优先级**：**先有稳定、可说明的单主线体验，再视需求与资源讨论分平台适配。**
